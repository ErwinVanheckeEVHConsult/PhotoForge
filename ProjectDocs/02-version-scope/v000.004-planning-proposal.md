# v0.4 Planning Proposal

---

## Context

- v0.1, v0.2, and v0.3 are implemented and released as alpha versions
- All v0.1–v0.3 behavior is frozen and must not change
- The backlog is clean and up to date
- All v0.4 scope must come strictly from backlog.md
- The existing architecture (scanner, planner, reporter, operations, CLI) must not be modified
- v0.4 must not introduce any changes to runtime behavior

---

## Candidate Scope (from backlog)

- Reusable project templates for planning, milestones, and releases
- Documentation for contributors
- Versioned releases and changelog

---

## Goal

Standardize the development workflow of PhotoForge by introducing a minimal, reusable, and strictly defined set of templates and supporting documentation.

This version formalizes how work is defined, executed, and released, without affecting the runtime system.

---

## Responsibilities

- Define a minimal and consistent template set for:
  - planning proposals
  - milestone definitions
  - release preparation
  - changelog entries

- Ensure all templates:
  - follow the exact structure and tone used in existing planning and milestone threads
  - are copy-paste ready without structural changes
  - enforce clear, deterministic work definitions

- Introduce minimal contributor documentation that:
  - explains how to use the templates
  - defines the expected workflow for planning, milestones, and releases
  - remains strictly procedural and concise

- Standardize release and changelog practices through templates only

---

## Non-Goals

- No changes to scanner, planner, reporter, operations, or CLI
- No changes to SPEC.md or versioned specifications
- No changes to runtime behavior or output
- No introduction of workflow automation (CI, generators, scripts, etc.)
- No packaging or distribution changes (e.g. PyPI)
- No test coverage expansion
- No redesign of backlog structure
- No general documentation beyond template usage

---

## Deliverables

- A `ProjectDocs/templates/` directory containing:
  - planning proposal template
  - milestone template
  - release checklist template
  - changelog entry template

- A minimal contributor guide (e.g. `CONTRIBUTING.md` or `ProjectDocs/contributing.md`) covering:
  - how to use each template
  - where templates are located
  - the workflow for:
    - planning a version
    - executing milestones
    - preparing a release

- A development workflow document (ProjectDocs/development-workflow.md) defining:
  - lifecycle steps
  - document storage
  - naming conventions
  - validation and release process

---

## Constraints

- All templates and documentation must be plain text or markdown
- No new tooling, frameworks, or dependencies
- No abstraction for hypothetical future use
- No duplication of existing documents unless converted into reusable templates
- Templates must be directly usable without interpretation
- Contributor documentation must remain minimal and procedural

---

## Acceptance Criteria

- A new version (v0.4.0-alpha1) can be fully planned, executed, and released using only the templates
- Future milestone threads can be created by copying the milestone template without structural changes
- Planning proposals follow a consistent structure across versions
- Release preparation is standardized and repeatable via checklist
- Changelog entries follow a consistent format
- Contributor documentation is sufficient to execute the workflow without external guidance
- No impact on runtime code or behavior

---

## Risks

- Overengineering templates beyond actual needs
- Introducing unnecessary structure or verbosity
- Creating templates that are not used in practice
- Blurring the boundary between templates and general documentation

---

## Success Definition

- Reduced friction when starting planning threads and milestones
- Consistent structure across all project artifacts
- Clear and repeatable release process
- No ambiguity in how work is defined and executed
- No impact on existing functionality
