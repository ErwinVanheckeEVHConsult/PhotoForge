# Milestone — v000.005-ms025 — metadata-extraction-extended

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic JPEG-only pipeline
- v0.5 introduces multi-format support, metadata normalization, and contextual grouping
- MS022 introduced deterministic detection of additional formats
- MS023 introduced deterministic format-specific validation boundaries
- MS024 introduced a passive metadata abstraction layer with exactly:
  - `timestamp`
  - `timestamp_source`
- Existing JPEG extraction behavior remains the reference behavior and must not change
- Scanner currently processes JPEG files through:
  - file stat access
  - `extract_timestamp(...)`
  - SHA-256 hashing
  - `FileRecord` construction
- Non-JPEG formats are currently recognized but not yet processed
- This milestone corresponds to `metadata-extraction-extended` in the approved milestone plan 0

---

## Goal

Extend metadata extraction to newly supported formats using deterministic, format-specific extraction paths that produce normalized metadata without changing existing JPEG behavior.

---

## Responsibilities

- Introduce format-specific metadata extraction for supported non-JPEG formats
- Produce normalized metadata using the existing metadata abstraction layer
- Preserve existing JPEG extraction behavior exactly
- Limit extraction to pipeline-relevant metadata only
- Keep format-specific logic outside the abstraction layer
- Prepare extracted metadata for later normalization and pipeline-integration milestones

---

## Deliverables

- `Format-specific metadata extraction for supported non-JPEG formats`
- `Deterministic metadata extraction paths per format`
- `Extraction functions produce values compatible with the normalized metadata structure`
- `JPEG metadata extraction remains behaviorally identical`
- `No changes to hashing, grouping, planning, or reporting behavior`

---

## Behavior

### Required behavior

- Metadata extraction must be extended for the following formats:
  - PNG
  - HEIC / HEIF
  - RAW formats
  - Video formats
- Extraction in this milestone is limited to:
  - `timestamp`
  - `timestamp_source`
- Existing JPEG extraction must continue to use the current EXIF fallback chain exactly:
  1. EXIF DateTimeOriginal
  2. EXIF DateTimeDigitized
  3. EXIF DateTime
  4. file modification time
- Non-JPEG extraction must be deterministic for identical input
- Format-specific extraction logic must remain outside the metadata abstraction layer
- Extracted values must be representable through the normalized metadata structure introduced in MS024
- Missing metadata must not be guessed or inferred
- No metadata enrichment is allowed
- Fallback to file modification time must be explicitly defined per format and must be deterministic
- `timestamp_source` values must be explicit and deterministic
- Extraction must rely only on stable, documented metadata fields provided by the chosen library
- Extraction must not rely on library behavior that is known to vary across versions or environments
- Extraction must not depend on environment-specific behavior (OS, locale, external tools)
- For identical extraction inputs, extraction results must be identical across runs
- Extraction inputs include:
  - file content
  - file extension / format path
  - file modification time when that format-specific path explicitly allows mtime fallback

### Format Rules

- PNG:
  - read existing metadata only if deterministically available through the chosen extraction path
  - otherwise use file modification time
- HEIC / HEIF:
  - read existing embedded timestamp metadata only if deterministically available through the chosen extraction path
  - otherwise use file modification time
- RAW:
  - read existing embedded timestamp metadata only if deterministically available through the chosen extraction path
  - otherwise use file modification time
- Video:
  - use embedded timestamp metadata only if deterministically available through the chosen extraction path
  - otherwise use file modification time

### Extraction Boundary

- Existing `extract_timestamp(...)` must remain unchanged
- JPEG extraction logic must not be rewritten or rerouted in this milestone
- New format-specific extractors may be introduced as separate functions or modules
- The metadata abstraction layer must remain structural, not format-specific

### Rules

- Deterministic behavior only
- No implicit or heuristic decisions
- No metadata inference or enrichment
- No additional metadata fields beyond `timestamp` and `timestamp_source`
- No behavioral changes to existing JPEG extraction
- No branching-based behavior changes outside defined format-specific extraction paths

---

## Constraints

- No changes to hashing logic
- No changes to duplicate grouping
- No changes to planner behavior
- No changes to reporter behavior
- No modification of `FileRecord` fields or structure
- No modification of the existing EXIF extraction logic in `extract_timestamp(...)`
- No changes to canonical selection logic
- No contextual grouping behavior introduced
- No metadata fields beyond `timestamp` and `timestamp_source`
- No abstraction-layer redesign
- No heuristic parsing of filenames or folder names
- No GPS extraction, keyword extraction, or sidecar support in this milestone
- Format-specific metadata extraction must not be integrated into the scanner or active pipeline in this milestone
- Non-JPEG formats must not produce FileRecord objects in this milestone
- Extraction functions must remain unused by the active pipeline

---

## Acceptance Criteria

- Deterministic metadata extraction exists for supported non-JPEG formats
- Extracted metadata is limited to:
  - `timestamp`
  - `timestamp_source`
- JPEG extraction produces identical results to prior behavior
- Existing `extract_timestamp(...)` behavior remains unchanged
- Format-specific extraction logic is clearly separated from the abstraction layer
- No additional metadata fields are introduced
- No changes occur in hashing, grouping, planning, or reporting behavior
- No inference or enrichment behavior is introduced
- Extracted timestamps and timestamp sources are reproducible for identical input

---

## Risks

- Format libraries may expose inconsistent or weak metadata
- Hidden non-determinism in third-party parsing behavior
- Accidental regression in JPEG extraction behavior
- Metadata extraction scope may expand implicitly beyond timestamp handling
- Format-specific extraction logic may leak into the abstraction layer

---

## Success Definition

- Supported non-JPEG formats have deterministic timestamp extraction paths
- Existing JPEG extraction remains strictly identical
- All extracted metadata maps cleanly to the normalized metadata structure
- No unintended side effects are introduced into the rest of the pipeline
- The codebase is ready for metadata normalization and later integration milestones
