# Milestone - v000.007-ms049 - timestamp-representation-model

## Type

implementation

---

## Context

- MS047 provides EXIF timestamp candidates with optional timezone offsets
- MS048 introduces a candidate-based timestamp resolution pipeline
- TimestampCandidate contains:
  - naive timestamp
  - optional timezone_offset
  - precision
- Downstream milestones (MS050+) require deterministic, comparable representations
- Requirement R007 defines the timestamp representation model

---

## Goal

Implement the timestamp representation model for:

- each valid timestamp candidate
- the resolved primary timestamp

The implementation must:

- generate all valid representations deterministically
- preserve strict separation of concerns
- enforce validity rules without implicit behavior

---

## Affected Components

- ``src/photoforge/metadata.py``
- ``src/photoforge/model.py`` (if representation structures are extended)

---

## Inputs

- ``TimestampCandidate``

---

## Outputs

- ``NormalizedMetadata``

Containing:

- ``naive_timestamp``
- ``timestamp_source``
- ``timezone_offset``
- ``timezone_aware_timestamp`` (optional)
- ``utc_timestamp`` (optional)

Timestamp_source must be derived from candidate.source_detail

---

## Processing Rules

### 1. Representation Types

The following representations may exist:

1. Naive representation
2. Timezone-aware representation
3. UTC representation

---

### 2. Naive Representation

- Always present
- Must equal ``candidate.naive_timestamp``
- Must not be modified

---

### 3. Timezone-Aware Representation

Created only if:

- ``timezone_offset`` is not ``None``

Rules:

- Must be created using:

  ``naive_timestamp.replace(tzinfo=timezone(offset))``

- Must correspond exactly to the naive timestamp
- Must not modify the underlying datetime value

---

### 4. UTC Representation

Created only if:

- Timezone-aware representation exists

Rules:

- Must be derived from timezone-aware timestamp:

  ``aware.astimezone(timezone.utc)``

- Must be deterministic
- Must not be created if timezone is missing

---

### 5. Invalid Conditions

The following must raise errors:

- naive timestamp has ``tzinfo`` set
- invalid timezone_offset
- timezone-aware timestamp must correspond exactly to naive timestamp
(i.e. removing tzinfo must yield the original naive timestamp)
- candidates with precision "date" must be rejected

---

### 6. No Implicit Behavior

- No timezone inference allowed
- No default timezone allowed
- No UTC creation without explicit offset

---

### 7. Determinism

- Same input candidate → identical normalized output
- No environment-dependent behavior
- No reliance on system timezone

---

## Invariants

- Normalization operates on a single candidate
- No cross-candidate logic
- No precedence logic
- No fallback logic
- No mutation of original candidate data
