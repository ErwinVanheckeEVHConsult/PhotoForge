# Milestone - v000.007-ms049 - timestamp-representation-model

## Type

implementation

---

## Context

- Current pipeline produces a single naive timestamp and a source string per file via `NormalizedMetadata`
- `metadata.py` enforces:
  - timestamp must be `datetime`
  - timestamp must be naive (no timezone)
  - no transformation or inference is allowed
- Timestamp extraction occurs in `metadata_extractors/*`
- Scanner orchestrates extraction → normalization → record creation
- No representation model currently exists for:
  - timezone-aware timestamps
  - UTC timestamps
- Requirements define explicit representation model constraints:
  - representations must be explicit
  - no implicit timezone inference
  - representations must only exist when valid

Affected components:

- `src/photoforge/metadata.py`
- `src/photoforge/model.py` (if timestamp structure is extended)
- `src/photoforge/metadata_extractors/*` (if additional data is surfaced)
- `src/photoforge/scanner.py` (if integration is required)

---

## Goal

Implement a deterministic timestamp representation model supporting:

- naive timestamp (existing)
- timezone-aware timestamp (only when explicitly available)
- UTC timestamp (only when explicitly derivable from valid timezone-aware input)

All representations must be:

- explicitly created
- mutually consistent
- never inferred implicitly

---

## Affected Components

- `src/photoforge/metadata.py`
  - extend normalization to support multiple representations
- `src/photoforge/model.py`
  - extend data model if required to store representations
- `src/photoforge/metadata_extractors/*`
  - may optionally expose timezone-aware timestamps if available
- `src/photoforge/scanner.py`
  - must integrate updated normalized metadata without altering routing behavior

---

## Behavior Specification

### Inputs

- `timestamp_naive: datetime`
- `timestamp_source: str`
- Optional:
  - `timestamp_aware: datetime` (tz-aware)
  - `timezone_offset: tzinfo`

Inputs must originate from extractors only.

---

### Processing Rules

1. Naive Timestamp
   - Must always be present
   - Must remain unchanged from extractor output
   - Must be validated as naive

2. Timezone-aware Timestamp
   - Must be created only if extractor provides valid timezone information
   - Must not be created by inference
   - Must correspond to the same instant as naive timestamp

3. UTC Timestamp
   - Must be created only if a valid timezone-aware timestamp exists
   - Must be computed via deterministic conversion
   - Must not be created from naive timestamp alone

4. Consistency Rules
   - If aware timestamp exists:
     - naive timestamp must equal aware timestamp with timezone removed
   - UTC timestamp must equal aware timestamp converted to UTC
   - No conflicting representations allowed

5. Invalid Data Handling
   - Invalid timezone → discard aware + UTC representations
   - Invalid timestamp → fail normalization
   - Partial data must not produce partial representations

6. No Inference Rule
   - No timezone may be inferred from:
     - filename
     - folder
     - filesystem
   - No default timezone allowed

---

### Outputs

A normalized metadata structure containing:

- `timestamp_naive: datetime` (required)
- `timestamp_source: str` (required)
- `timestamp_aware: datetime | None`
- `timestamp_utc: datetime | None`

All fields must be deterministic and immutable.

---

### Edge Cases

- Extractor returns naive only → aware=None, utc=None
- Extractor returns aware but invalid tz → drop aware + utc
- Extractor returns inconsistent naive vs aware → fail
- Timezone offset present but timestamp missing → fail
- DST transitions → handled only via provided tzinfo (no custom logic)

---

## Invariants

- Existing validated behavior must remain unchanged when only naive timestamp is present
- No implicit timezone inference is introduced
- Output structure must remain deterministic
- Extraction logic remains outside `metadata.py`

- Given invalid timezone → aware and UTC are not created
- Given inconsistent inputs → normalization fails
- UTC conversion is deterministic and correct
- No timezone inference occurs under any condition
- Existing behavior (naive-only pipeline) remains unchanged
- Output is identical across runs for identical inputs
