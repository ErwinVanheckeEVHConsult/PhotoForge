## Behavior Specification

### Inputs

- `<folder_name>` as string (name of the immediate parent directory only)

---

### Processing Rules

- Extraction must operate on folder name only (no path, no filename)
- Parsing must follow R011 exactly
- Parsing must be left-to-right
- First valid match must be used
- Only one candidate may be produced
- A match must not be part of a longer numeric sequence
- A valid match must be delimited by:
  - start or end of string
  - or a non-numeric character
- Matches must respect character boundaries
- Invalid matches must be rejected
- Parsing must be deterministic and independent of environment or locale

When multiple matches exist:

- The earliest match position must be selected
- If multiple matches start at the same position:
  - range patterns take precedence over single-date patterns

---

### Supported Patterns (as defined in R011)

Single date:

- YYYYMMDD
- YYYY-MM-DD

Date range:

- YYYYMMDD-YYYYMMDD
- YYYY-MM-DD - YYYY-MM-DD

---

### Outputs

- ``tuple[TimestampCandidate, ...]``

At most one candidate may be returned.

Each candidate must contain:

- ``source_kind = "folder"``
- ``source_detail`` = pattern identifier (see below)
- ``naive_timestamp`` = parsed datetime
- ``precision = "date"``
- ``timezone_offset = None``

All folder-derived timestamps are date-based:

- time must be set to ``00:00:00``

For date range patterns:

- only the **start date** must be used

---

### Source Detail Mapping

Each pattern must map to a deterministic source_detail:

- YYYYMMDD → folder_yyyymmdd
- YYYY-MM-DD → folder_yyyy-mm-dd
- YYYYMMDD-YYYYMMDD → folder_yyyymmdd_range
- YYYY-MM-DD - YYYY-MM-DD → folder_yyyy-mm-dd_range

---

### Edge Cases

- Multiple matches → first valid match is used
- Invalid numeric values → reject match
- Mixed patterns → only first valid match used
- No valid pattern → no timestamp produced
- Overlapping digit sequences → must respect boundary rules
- Range patterns → end date ignored for timestamp value
