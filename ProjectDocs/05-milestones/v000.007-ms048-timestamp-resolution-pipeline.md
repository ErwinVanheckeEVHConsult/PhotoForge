## Context

- Metadata extraction layer is defined (MS046)
- EXIF extraction produces timestamp candidates (MS047)
- Multiple timestamp sources exist and must be evaluated together
- Current model supports only a single timestamp, which is insufficient for:
  - precedence (R005)
  - validity-based fallback (R006)
  - comparison and diagnostics (R008, R009, R013)
- Architecture requires strict separation:
  - extractors = source-local parsing
  - resolution = cross-source logic
  - metadata.py = normalization only

---

## Goal

Introduce a deterministic timestamp resolution layer that:

- consumes timestamp candidates from all sources
- validates candidates according to R006
- resolves a single primary timestamp according to R005
- preserves valid candidates for downstream processing

## Behavior Specification

### Inputs

- tuple of timestamp candidates from all extractors

Each candidate contains:

- source kind
- naive timestamp
- optional timezone offset

---

### Processing Rules

#### 1. Candidate Collection

- Collect all candidates from:
  - EXIF
  - filename
  - folder
  - filesystem
- Extractors must not perform cross-source logic

#### 2. Candidate Validation

A candidate is valid only if:

- it can be parsed deterministically
- it satisfies calendar constraints
- it contains full datetime components:
  - year
  - month
  - day
  - hour
  - minute
  - second

Date-only candidates are considered invalid for resolution.

Invalid candidates:

- must be discarded
- must not block evaluation of other candidates

#### 3. Precedence

Apply strict precedence:

1. EXIF
2. filename
3. folder
4. filesystem

#### 4. Resolution

- Select the first valid candidate by precedence
- This becomes the primary timestamp

#### 5. Preservation

- All valid candidates must be preserved
- Must be available for:
  - comparison (MS050)
  - diagnostics (MS050, MS055)

#### 6. Determinism

- Ordering must be explicit and fixed
- No environment-dependent behavior
- Identical input → identical output

---

### Outputs

- primary timestamp candidate
- tuple of valid timestamp candidates

---

### Edge Cases

- No valid candidates → error condition
- Invalid higher-precedence candidate → ignored
- Multiple valid candidates → only highest-precedence selected
