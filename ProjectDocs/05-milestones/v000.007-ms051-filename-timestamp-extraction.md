## Behavior Specification

### Inputs

- `<filename>` as string (full filename including extension)

### Processing Rules

- Extraction must operate on filename only (no path, no folder context)
- Parsing must follow R010 exactly
- Parsing must be left-to-right
- First full match must be used
- Full datetime patterns must have priority over date-only patterns
- A match must not be part of a longer numeric sequence
- A valid match must be delimited by:
  - start or end of string
  - or a non-numeric character
- Matches must respect character boundaries
- Invalid matches must be rejected
- Parsing must be deterministic and independent of environment or locale

### Supported Patterns (as defined in R010)

Full datetime:

- YYYYMMDD_HHMMSS
- YYYY-MM-DD_HH-MM-SS
- YYYY-MM-DD HH-MM-SS

Date-only:

- YYYYMMDD
- YYYY-MM-DD

### Outputs

- ``tuple[TimestampCandidate, ...]``

At most one candidate may be returned.

Each candidate must contain:

- ``source_kind = "filename"``
- ``source_detail`` = pattern identifier (see below)
- ``naive_timestamp`` = parsed datetime
- ``precision``:
  - "datetime" for full patterns
  - "date" for date-only patterns
- ``timezone_offset = None``

Date-only timestamps must default to time ``00:00:00``.

### Source Detail Mapping

Each pattern must map to a deterministic source_detail:

- YYYYMMDD_HHMMSS → filename_yyyymmdd_hhmmss
- YYYY-MM-DD_HH-MM-SS → filename_yyyy-mm-dd_hh-mm-ss
- YYYY-MM-DD HH-MM-SS → filename_yyyy-mm-dd_hh-mm-ss_space
- YYYYMMDD → filename_yyyymmdd
- YYYY-MM-DD → filename_yyyy-mm-dd

### Edge Cases

- Multiple matches → first valid full match is used
- Invalid numeric values → reject match
- Mixed patterns → only first valid match used
- No valid pattern → no timestamp produced
- Overlapping digit sequences → must respect boundary rules
