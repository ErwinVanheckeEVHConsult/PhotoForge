## Model Definition

### Existing record model boundary

The contextual grouping model operates on existing `FileRecord` objects only.

`FileRecord` is outside the scope of this milestone and must remain unchanged.

The contextual grouping layer must not:

- modify `FileRecord`
- wrap `FileRecord`
- duplicate `FileRecord`
- extend `FileRecord`
- embed group identifiers into `FileRecord`

---

### Member reference model

A contextual group must reference files indirectly through a stable record reference.

For this milestone, the stable record reference is defined as:

- the normalized absolute file path represented as a string

This reference is called `record_ref`.

Rules:

- `record_ref` must uniquely identify one `FileRecord` within a given scan result
- `record_ref` must be derived directly from the normalized `FileRecord.path`
- `record_ref` must correspond to the normalized absolute path produced by the scanner normalization step
- `record_ref` must be representable as a string
- `record_ref` must not depend on scan order
- `record_ref` must not depend on object identity or memory location
- `record_ref` must not require mutation of `FileRecord`
- All `record_ref` values within a grouping instance must be unique

### Concrete record_ref conversion rule

- `record_ref` must be produced as `str(file_record.path)`
- `file_record.path` must already be the normalized absolute path produced by the scanner

### Concrete group_id rule

- `group_id` must be computed from a delimiter-safe encoding of `member_refs`
- The encoding must be unambiguous for all valid path values
- The encoding of `member_refs` must be canonical and deterministic
- The encoding must be defined as:
  - JSON array of strings using UTF-8 encoding
  - without whitespace variation
- `group_id` must be computed as:
  - SHA-256 hex digest of the UTF-8 encoded canonical JSON representation of `member_refs`
- Each contextual group within a grouping instance must have a unique `group_id`
- Contextual grouping structures must be immutable once constructed

### Instance invariant rule

For any concrete `ContextualGrouping` instance produced by grouping logic:

- every `FileRecord` in scope must appear in exactly one contextual group
- no `FileRecord` may appear in more than one contextual group
- no contextual group may contain zero members

---

### Contextual group structure

A contextual group is defined as a structural object with the following fields:

- `group_id: str`
- `member_refs: tuple[str, ...]`

Meaning:

- `group_id` is the deterministic identifier of the contextual group
- `member_refs` is the complete ordered tuple of `record_ref` values belonging to the group

---

### Contextual grouping container structure

The full contextual grouping result is defined as a structural object with the following fields:

- `groups: tuple[ContextualGroup, ...]`

Meaning:

- `groups` is the complete ordered tuple of contextual groups for a given set of `FileRecord` objects

---

### Illustrative structural form

The model defined by this milestone corresponds to the following structure:

```python
@dataclass(frozen=True)
class ContextualGroup:
    group_id: str
    member_refs: tuple[str, ...]


@dataclass(frozen=True)
class ContextualGrouping:
    groups: tuple[ContextualGroup, ...]
```

## Behavior

### Required behavior

- Contextual grouping must be defined as a layer separate from duplicate grouping
- Contextual grouping must be defined as a layer separate from canonical selection
- Contextual grouping must not be derived from or reuse duplicate grouping identifiers or structures
- Contextual grouping must reference existing `FileRecord` objects indirectly through `record_ref`
- The model must be capable of representing a partition of the processed `FileRecord` set
- Group membership must be represented only by `member_refs`
- The grouping model must be representable independently of pipeline execution
- The grouping model must not alter scanning, hashing, duplicate grouping, canonical selection, planning, reporting, or execution behavior
- The grouping model must be deterministic for identical inputs
- No grouping construction logic may be defined in this milestone
- No grouping rules may be defined in this milestone
- No threshold, heuristic, ranking, scoring, or inference logic may be defined in this milestone

---

### Partition rules

- Coverage: every `FileRecord` in scope must be representable within the grouping model
- Exclusivity: no `FileRecord` may appear in more than one contextual group
- Non-emptiness: every contextual group must contain at least one member
- Closure: all `member_refs` must refer to existing `FileRecord` objects

---

### Empty input rule

- If no `FileRecord` objects exist, `groups` must be an empty tuple

---

### Ordering rules

- `member_refs` must preserve the deterministic ordering defined by the grouping construction rules in a later milestone
- `member_refs` must be sorted lexicographically
- Future grouping logic must produce member sets that, when sorted lexicographically, preserve deterministic identity
- The ordering of `member_refs` must be stable and deterministic
- `groups` must be sorted lexicographically by `group_id`
- No ordering may depend on scan order or filesystem traversal

---

### Identifier rules

- `group_id` must be deterministic
- `group_id` must be derived only from `member_refs`
- `group_id` must be derived from the ordered sequence of `member_refs`
- `group_id` must be stable for identical group membership
- `group_id` must not depend on external state or ordering outside of `member_refs`

---

### Metadata input rules

Only the following metadata inputs are allowed in this version:

- `timestamp`
- `timestamp_source`

No other metadata is allowed.
