# Milestone — v000.005-ms030 — contextual-grouping-integration

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic pipeline for scanning, hashing, duplicate grouping, and planning
- v0.5 introduces metadata normalization and contextual grouping
- MS028 defined a complete formal structural model for contextual grouping
- MS029 (previous milestone) defines deterministic grouping rules and group construction logic
- Contextual grouping currently exists only as a model and (after MS029) as a standalone computation
- The active pipeline does not yet include contextual grouping
- Existing pipeline behavior must remain strictly unchanged

---

## Goal

Integrate contextual grouping into the pipeline as a deterministic, side-effect-free structural output without modifying any existing pipeline behavior or outputs.

---

## Responsibilities

- Integrate contextual grouping computation into the pipeline
- Ensure grouping is computed deterministically from existing `FileRecord` objects
- Ensure contextual grouping is produced as a separate structural output
- Ensure no impact on existing scanner, planner, reporter, or CLI behavior
- Preserve full determinism and reproducibility
- Define and implement one internal orchestration function as the sole integration point for contextual grouping
- Introduce a dedicated orchestration module responsible for pipeline composition

---

## Deliverables

- `Contextual grouping computed from FileRecord set`
- `ContextualGrouping object produced as part of pipeline flow`
- `No changes to existing pipeline outputs or behavior`
- `No changes to PlanResult structure`

---

## Behavior

### Required behavior

- Contextual grouping must be computed from the complete set of valid `FileRecord` objects
- Grouping must use only:
  - `timestamp`
  - `timestamp_source`
- Grouping must follow the deterministic rules defined in the previous milestone
- Contextual grouping must be computed exactly once per pipeline execution
- Contextual grouping must be fully deterministic for identical input

### Integration rules

- Contextual grouping must be computed after scanning and metadata normalization
- Contextual grouping must not modify or filter the `FileRecord` set
- Contextual grouping must not influence:
  - duplicate grouping (hash-based)
  - canonical selection
  - planner behavior
  - action generation
  - reporting output (existing formats)

- Contextual grouping must not be used to:
  - change filenames
  - change folder structure
  - change grouping of duplicates
  - influence any existing decision logic

- Contextual grouping must be invoked only by the internal orchestration function defined in this milestone
- Neither `scanner.py` nor planner logic may compute contextual grouping directly
- Within the orchestration function, contextual grouping must be computed once from the complete valid `tuple[FileRecord, ...]` and then propagated unchanged

- The orchestration function must be implemented in a dedicated module:
  - `pipeline.py`

- The orchestration module is the only location where:
  - scanning
  - planning
  - contextual grouping
  are composed into a single flow

- `pipeline.py` must not implement:
  - scanning logic
  - planning logic
  - grouping rules

- `pipeline.py` must only orchestrate calls to existing components

### Output rules

- Contextual grouping must be produced as a separate object:
  - `ContextualGrouping`
- The grouping output must not be embedded into:
  - `FileRecord`
  - `PlannedRecord`
  - `PlanResult`

- Contextual grouping must be integrated through a new internal orchestration interface

- The orchestration interface must:
  - consume the existing valid `tuple[FileRecord, ...]`
  - invoke the existing planning flow unchanged
  - compute `ContextualGrouping` once from the complete valid `FileRecord` set
  - return exactly:
    - `PlanResult`
    - `ContextualGrouping`

- CLI output must remain unchanged in this milestone

---

## Constraints

- No changes to scanner behavior
- No changes to hashing logic
- No changes to duplicate grouping
- No changes to canonical selection
- No changes to planner behavior
- No changes to reporter behavior
- No changes to CLI behavior
- No modification of `FileRecord`
- No modification of `PlannedRecord`
- No modification of `PlanResult`
- No integration into JSON or console output
- No metadata beyond allowed inputs
- No redesign introduced
- `scan_directory(...)` must remain unchanged
- Existing planning entry points must remain unchanged
- `PlanResult` must remain unchanged
- No contextual grouping logic may be implemented directly inside `scanner.py`
- No contextual grouping logic may be implemented directly inside planner decision logic
- No orchestration logic may be implemented in:
  - `scanner.py`
  - planner implementation modules
  - CLI layer
- CLI must invoke the orchestration function but must not implement orchestration logic

---

## Acceptance Criteria

- Contextual grouping is computed deterministically from `FileRecord` objects
- Contextual grouping satisfies all structural invariants defined in MS028
- Contextual grouping satisfies all grouping rules defined in MS029
- Contextual grouping does not modify or influence any existing pipeline outputs
- For identical input:
  - identical `FileRecord` → identical contextual grouping
  - identical contextual grouping → identical `group_id` values
- Existing outputs remain strictly identical:
  - `records`
  - `actions`
  - `corrupt_files`
  - console output
  - JSON output
- No additional fields are introduced in existing models
- No unintended side effects occur
- A single internal orchestration function is defined as the sole integration point for contextual grouping
- The orchestration function returns exactly:
  - `PlanResult`
  - `ContextualGrouping`
- `scan_directory(...)` remains unchanged
- Existing planning entry points remain unchanged
- Contextual grouping is computed once from the complete valid `FileRecord` set and propagated unchanged

---

## Risks

- Accidental coupling between contextual grouping and duplicate grouping
- Hidden influence on planner behavior
- Introduction of implicit ordering dependencies
- Leakage of grouping data into existing models
- Violation of determinism guarantees

---

## Success Definition

- Contextual grouping is fully integrated as a passive structural output
- Existing pipeline behavior remains strictly unchanged
- Determinism is preserved end-to-end
- The system is ready for future milestones that may expose or use contextual grouping
