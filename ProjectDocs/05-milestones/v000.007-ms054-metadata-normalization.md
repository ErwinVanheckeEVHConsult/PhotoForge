## Type

implementation

## Context

- Extraction layer (MS047, MS051, MS052, MS053) produces multiple TimestampCandidate objects from heterogeneous sources
- Timestamp resolution (MS048) selects a primary candidate and preserves valid candidates
- Timestamp representation (MS049) defines naive / aware / UTC representations
- Timestamp comparison and diagnostics (MS050) operate on candidate sets
- The pipeline now produces a structured resolution result, but normalization into final metadata output must be explicitly defined
- metadata.py must remain responsible for normalization only

Affected components:

- src/photoforge/metadata.py
- src/photoforge/model.py (read-only usage)
- src/photoforge/timestamp_resolution.py (read-only usage)

---

## Goal

Normalize the resolved timestamp candidate into a deterministic, final metadata structure suitable for FileRecord construction, without introducing new logic or altering resolution behavior.

---

## Affected Components

- `src/photoforge/metadata.py` (implementation)

Untouched:

- `metadata_extractors/*`
- `timestamp_resolution.py`
- `timestamp_diagnostics.py`
- `scanner.py` (no logic changes)

---

## Behavior Specification

### Inputs

- `primary_candidate: TimestampCandidate`

---

### Processing Rules

- Normalization must operate on the resolved primary candidate only
- No extraction logic is allowed
- No resolution logic is allowed
- No comparison or diagnostics logic is allowed
- No fallback logic is allowed
- Normalization must be deterministic

---

### Representation Handling

Normalization must derive the canonical timestamp from the primary candidate using MS049 rules.

Representation construction must remain implicit:

- If timezone_offset is present:
  - UTC timestamp must be derived and used as canonical timestamp
- Else:
  - naive timestamp must be used as canonical timestamp

No intermediate representations must be stored or exposed.

---

### Outputs

- NormalizedMetadata object

The object must contain:

- `timestamp` → selected canonical timestamp (see below)
- `timestamp_source` → derived from candidate (source_detail)

---

### Canonical Timestamp Selection

- If UTC representation exists → use UTC timestamp
- Else → use naive timestamp

---

### Source Mapping

- `timestamp_source` must equal `primary_candidate.source_detail`

---

### Edge Cases

- Missing timezone_offset → naive-only representation
- Invalid timezone_offset → must not occur (validated earlier)
- Primary candidate always present (guaranteed by MS048)

---

## Invariants

- No modification of candidate data
- No re-evaluation of precedence
- No introduction of new candidates
- No diagnostics logic
- Deterministic output for identical input

---

## Responsibilities

- Implement normalization logic in metadata.py
- Construct final metadata structure from primary candidate
- Ensure consistency with representation rules
- Ensure deterministic behavior

---

## Deliverables

- Implementation in `metadata.py`
- Deterministic normalization behavior

---

## Constraints

- No changes outside defined scope
- No extraction logic
- No resolution logic
- No diagnostics logic
- No implicit behavior

---

## Validation Criteria

- Given naive-only candidate → naive timestamp returned
- Given aware candidate → UTC timestamp returned
- timestamp_source matches source_detail
- No regression in pipeline behavior
- Deterministic output across runs
- Implementation matches MS049 representation rules
