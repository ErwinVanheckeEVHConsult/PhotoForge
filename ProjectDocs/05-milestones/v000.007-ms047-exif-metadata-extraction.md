# Milestone - v000.007-ms047 - exif-metadata-extraction

## Type

implementation

---

## Context

- Metadata extraction layer structure is defined in MS046
- EXIF extraction must conform to the metadata extraction layer
- EXIF extraction must integrate into:

  ``src/photoforge/metadata_extractors/``

- EXIF extraction must support:
  - multiple timestamp candidates
  - corresponding timezone extraction
- Current implementation does not extract EXIF timezone information
- Requirements R003 and R004 define deterministic EXIF timestamp and timezone behavior

---

## Goal

Implement EXIF metadata extraction aligned with the metadata extraction layer, including:

- extraction of EXIF timestamp candidates (0..n)
- extraction of corresponding EXIF timezone offsets
- correct association between timestamp fields and their timezone fields
- no cross-source logic
- no implicit timezone inference

---

## Affected Components

- ``src/photoforge/metadata_extractors/exif.py``
- ``src/photoforge/metadata_extractors/__init__.py``
- ``src/photoforge/scanner.py`` (integration only, no logic changes)

## Inputs

- ``path: Path``

---

## Outputs

- ``tuple[TimestampCandidate, ...]``

Each candidate contains:

- ``source_kind = "exif"``
- ``source_detail`` = EXIF field identifier
- ``naive_timestamp`` = parsed EXIF datetime
- ``precision = "datetime"``
- ``timezone_offset`` = corresponding EXIF timezone offset or ``None``

---

## Processing Rules

### 1. EXIF Field Mapping

Timestamp fields:

- ``DateTimeOriginal`` (tag 36867)
- ``DateTimeDigitized`` (tag 36868)
- ``DateTime`` (tag 306)

Timezone fields:

- ``OffsetTimeOriginal`` (tag 36881)
- ``OffsetTimeDigitized`` (tag 36882)
- ``OffsetTime`` (tag 36880)

---

### 2. Timestamp Parsing

- Values must be parsed using:

  ``"%Y:%m:%d %H:%M:%S"``

- Invalid values must be ignored
- Parsing must be deterministic

---

### 3. Timezone Parsing

- Timezone values must follow EXIF format:

  ``±HH:MM``

- Parsed as ``timedelta``

- Invalid timezone values must be ignored

---

### 4. Field Association

Timezone must be applied only to the corresponding timestamp field:

| Timestamp Field       | Timezone Field           |
|-----------------------|--------------------------|
| DateTimeOriginal      | OffsetTimeOriginal       |
| DateTimeDigitized     | OffsetTimeDigitized      |
| DateTime              | OffsetTime               |

Rules:

- If timezone exists and is valid → attach to candidate
- If timezone is missing or invalid → ``timezone_offset = None``
- Timezone from one field must never be applied to another field

---

### 5. Candidate Creation

For each valid timestamp:

- create one ``TimestampCandidate``

- include:
  - parsed datetime
  - associated timezone offset (if valid)

- multiple candidates may be returned

---

### 6. Error Handling

- EXIF read failures must return empty candidate set
- Invalid fields must be ignored
- Extraction must not raise exceptions

---

### 7. Determinism

- Candidate order must follow field precedence:

  1. DateTimeOriginal
  2. DateTimeDigitized
  3. DateTime

- Ordering must be explicit and stable

---

## Invariants

- Extractor performs only local parsing and validation
- No cross-source logic
- No precedence resolution across sources
- No implicit timezone inference
- Naive timestamp must remain unchanged
- Timezone offset must not modify naive timestamp
