# Milestone — v000.006-ms042 — system-documentation-structure

## Type

documentation

---

## Context

* The workflow enforces that all workflow-governed documents must have a defined storage location
* Existing workflow steps define storage for:

  * backlog
  * version scope
  * scope approval
  * requirements
  * milestones
  * version documentation
  * release validation
  * release
  * workflow
  * templates
* Several documentation artifacts exist that do not belong to any defined workflow step:

  * system-level technical documentation (e.g. scanner, pipeline, model, metadata)
  * validation specifications (e.g. documentation-validation)
* These artifacts currently have no defined place in the workflow structure
* This creates a structural inconsistency:

  * storage rules are strict
  * but not all artifacts are covered

---

## Goal

Define a deterministic, complete storage model for system-level documentation that:

* assigns a single authoritative location for all such artifacts
* integrates with the workflow without modifying the step transition model
* removes all ambiguity about where non-step documentation must be stored

---

## Affected Documents

* `ProjectDocs/development-workflow.md`

---

## Definitions

### System Documentation Domain

A category of documentation that:

* describes system structure, behavior, or components
* is not created as part of a specific workflow step
* is not version-cycle scoped
* is not tied to milestone execution

Includes:

* architecture documentation
* module-level documentation
* pipeline and processing documentation
* data model documentation
* validation specifications

---

### Storage Location

All system documentation must be stored in:

* `ProjectDocs/00-system-documentation/`

Rules:

* This folder is part of the workflow-governed structure
* It is not associated with a numbered workflow step
* It exists outside the sequential execution flow

---

### Classification Rules

A document belongs to the system documentation domain if:

* it does not correspond to a workflow step output
* it is not version-specific
* it does not represent a milestone, requirement, or release artifact
* it describes system structure, behavior, or validation rules

A document must not be stored in this domain if it:

* is generated as part of a workflow step
* is version-scoped
* is defined by a template tied to a workflow step

---

### Maintenance Rules

System documentation:

* must remain consistent with implementation
* must remain consistent with requirements and workflow definitions
* must not contain placeholders
* must not contain ambiguous or implicit definitions

Updates to system documentation must occur when:

* implementation changes affect described behavior
* validation rules change
* structure or architecture is modified

---

## Invariants

* Existing workflow steps and their storage rules must remain unchanged
* Workflow transition model must remain unchanged
* Template enforcement rules must remain unchanged
* No existing document classification may be weakened or made ambiguous

---

## Responsibilities

* Define the system documentation domain explicitly
* Define its storage location
* Define classification rules
* Ensure no overlap with existing workflow step outputs
* Eliminate all ambiguity regarding orphan documentation

---
