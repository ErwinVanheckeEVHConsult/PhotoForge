# Milestone — v000.005-ms022 — scanner-multi-format-detection

Starting milestone.

---

## Context

- v0.1, v0.2, v0.3 are complete and released (alpha)
- v0.4 introduces workflow standardization only (no runtime changes)
- v0.5 introduces multi-format support, metadata layer, and contextual grouping
- Existing scanner currently processes JPEG files only (.jpg, .jpeg)
- File filtering is deterministic and extension-based
- Existing corrupt file classification behavior remains unchanged in this milestone
- Existing pipeline behavior must remain unchanged for JPEG files

---

## Goal

Extend the scanner to deterministically detect and include additional file formats without introducing any content processing or behavioral changes.

---

## Responsibilities

- Extend file extension filtering logic to include:
  - PNG
  - HEIC / HEIF
  - RAW formats (CR2, NEF, ARW, etc.)
  - Video formats (MP4, MOV)
- Ensure deterministic inclusion of supported formats
- Ensure deterministic exclusion of unsupported formats
- Preserve existing behavior for JPEG files exactly

---

## Deliverables

- ``Extended file extension support in scanner``
- ``Deterministic inclusion/exclusion logic for new formats``
- ``No changes to existing JPEG processing behavior``

---

## Behavior

### Required behavior

- Scanner must include files with supported extensions (case-insensitive)
- Supported extensions must include:
  - .jpg, .jpeg (existing)
  - .png
  - .heic, .heif
  - .cr2, .nef, .arw (explicit RAW formats supported in this milestone)
  - .mp4, .mov
- All other file types must be ignored
- File detection must remain extension-based only
- No file content must be read or interpreted in this milestone
- Non-JPEG formats are passed through the pipeline without format-specific handling
- Files must only be classified as corrupt if deterministic processing fails (e.g. I/O failure, unreadable file)
- Lack of format-specific support must not result in corrupt classification
- No format-specific handling is introduced in this milestone

### Rules

- Deterministic behavior only
- No implicit or heuristic decisions
- No deviation from defined structure
- Timestamp extraction behavior remains unchanged in this milestone; non-JPEG formats may fall back to mtime under the existing extraction logic
- Supported extensions list is closed and must not be expanded within this milestone
- Extension matching must be performed using normalized lowercase suffix comparison

---

## Constraints

- No metadata extraction changes
- No hashing changes
- No duplicate grouping changes
- No planner changes
- No reporter changes
- Corrupt file classification logic must remain unchanged for processable formats
- Newly recognized formats must not be classified as corrupt solely due to lack of processing support
- Files must only be classified as corrupt if they fail deterministic processing (e.g. I/O failure)
- No content-based validation
- No new data structures introduced

---

## Acceptance Criteria

- Scanner detects all supported file types deterministically
- JPEG behavior remains unchanged
- Unsupported files are still ignored
- No changes in downstream pipeline behavior
- No regression in existing functionality

---

## Risks

- Incorrect extension handling (case sensitivity issues)
- Inclusion of unintended file types
- Accidental introduction of content-based logic
- Regression in existing JPEG detection behavior

---

## Success Definition

- Scanner supports additional file formats through extension-based detection
- Existing behavior for JPEG files remains identical
- No impact on downstream pipeline components
- Detection remains deterministic and minimal
