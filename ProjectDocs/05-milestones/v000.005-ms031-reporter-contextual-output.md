# Milestone — v000.005-ms031 — reporter-contextual-output

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic reporting system for planned actions
- Reporter currently consumes `PlanResult` and produces:
  - console output
  - JSON output (optional)
- Current reporter entry points are:
  - `render_console_report(plan_result)`
  - `render_json_report(plan_result)`
- MS028 defined the contextual grouping structural model
- MS029 defined deterministic grouping rules
- MS030 integrated contextual grouping into the pipeline via a dedicated orchestration layer
- Contextual grouping is now available as a passive structural output (`ContextualGrouping`)
- Reporter does not yet expose contextual grouping

---

## Goal

Extend the reporter to expose contextual grouping information in a deterministic and non-intrusive way without modifying existing output behavior unless explicitly requested.

---

## Responsibilities

- Extend reporter to support contextual grouping output
- Define explicit reporter function signatures for contextual grouping input
- Ensure grouping output is derived only from `ContextualGrouping`
- Preserve all existing reporting behavior by default
- Introduce explicit opt-in mechanism for contextual grouping output
- Define explicit CLI handoff of contextual grouping into reporter
- Maintain strict determinism and reproducibility

---

## Deliverables

- `Contextual grouping output support in reporter`
- `Explicit reporter interface for contextual grouping input`
- `Explicit CLI-to-reporter propagation of contextual grouping`
- `Explicit opt-in mechanism for contextual output`
- `Deterministic contextual grouping rendering`
- `No changes to default output behavior`

---

## Behavior

### Required behavior

- Reporter must accept `ContextualGrouping` as an additional input
- Reporter must not compute or modify contextual grouping
- Reporter must treat contextual grouping as read-only input
- Contextual grouping must not alter:
  - existing report structure
  - existing summary values
  - existing action output

### Reporter interface contract

Reporter function signatures must be extended to accept:

- `contextual_grouping: ContextualGrouping | None`
- `include_context: bool`

The reporter interface must be defined as:

```python
def render_console_report(
    plan_result: PlanResult,
    contextual_grouping: ContextualGrouping | None = None,
    include_context: bool = False,
) -> str
```

```python
def render_json_report(
    plan_result: PlanResult,
    contextual_grouping: ContextualGrouping | None = None,
    include_context: bool = False,
) -> str
```

Rules:

- Existing reporter functions must remain the canonical rendering entry points
- No duplicate contextual-only reporter functions may be introduced
- When `include_context` is `False`, reporter output must remain strictly identical to prior behavior
- When `include_context` is `True`, `contextual_grouping` must be provided
- Reporter must not infer contextual grouping presence from any source other than explicit function arguments

---

### Output modes

#### Default behavior

- Default console output must remain strictly unchanged
- Default JSON output must remain strictly unchanged
- Contextual grouping must not appear unless explicitly enabled

#### Opt-in behavior

- A new CLI flag must be introduced:

```bash
--context
```

- When `--context` is provided:
  - contextual grouping must be included in output
  - existing output must remain present and unchanged

---

### CLI propagation rules

- CLI must call the orchestration function introduced in the previous milestone
- CLI must receive exactly:
  - `PlanResult`
  - `ContextualGrouping`
- CLI must pass both objects to reporter functions
- CLI must pass `include_context=True` only when `--context` is provided
- CLI must pass `include_context=False` when `--context` is not provided
- CLI must not compute, transform, or validate contextual grouping
- CLI must not render contextual grouping directly
- CLI must remain a propagation layer only

---

### Console output rules

When `include_context=True`:

- Contextual grouping must be rendered as a separate section
- The contextual grouping section must appear after existing report output
- Each group must include:
  - `group_id`
  - ordered list of `record_ref` values

Rules:

- Output must be deterministic
- Groups must be printed in the exact order defined by `ContextualGrouping.groups`
- Members must be printed in the exact order defined in `member_refs`
- No additional formatting or interpretation is allowed
- Existing console output content must remain unchanged outside the added contextual section

---

### JSON output rules

When `include_context=True`:

- JSON output must include a new top-level field:

```json
"contextual_groups"
```

- Each group must be represented as:

```json
{
  "group_id": "<group_id>",
  "member_refs": ["<record_ref>", "..."]
}
```

Rules:

- Order of groups must match `ContextualGrouping.groups`
- Order of `member_refs` must be preserved
- No additional fields are allowed
- Existing JSON schema must remain unchanged otherwise

When `include_context=False`:

- The `contextual_groups` field must not be present

---

### Determinism rules

- For identical input:
  - identical `PlanResult` and identical `ContextualGrouping` must produce identical output
- Output must not depend on:
  - scan order
  - filesystem traversal
  - runtime environment
  - CLI invocation order

---

### Separation rules

- Reporter must not:
  - recompute grouping
  - validate grouping beyond structural correctness
  - interpret grouping semantics
- Reporter must not influence:
  - planner
  - pipeline orchestration
  - grouping logic

---

## Constraints

- No changes to scanner behavior
- No changes to planner behavior
- No changes to grouping logic
- No changes to `FileRecord`
- No changes to `PlannedRecord`
- No changes to `PlanResult`
- No changes to default CLI output
- No redesign introduced

---

## Acceptance Criteria

- Reporter accepts `ContextualGrouping` as input through the defined function signatures
- Reporter accepts `include_context: bool` through the defined function signatures
- Existing reporter functions remain the canonical rendering entry points
- CLI calls the orchestration function and receives:
  - `PlanResult`
  - `ContextualGrouping`
- CLI passes both objects to reporter functions
- CLI enables contextual output only when `--context` is provided
- Default output remains strictly identical for identical input
- `--context` enables contextual grouping output
- Console output includes contextual grouping in a separate deterministic section
- JSON output includes `contextual_groups` only when `include_context=True`
- Output ordering is fully deterministic
- No unintended side effects occur
- No existing output fields are modified or removed

---

## Risks

- Accidental modification of default output behavior
- Non-deterministic ordering in contextual output
- Leakage of grouping logic into reporter
- Schema drift in JSON output
- Ambiguous CLI-to-reporter propagation

---

## Success Definition

- Contextual grouping is visible to users only when explicitly requested
- Default behavior remains unchanged
- Reporter interface is fully explicit and unambiguous
- CLI propagation is fully explicit and unambiguous
- Output remains deterministic and reproducible
- Reporter cleanly exposes contextual grouping without affecting pipeline logic
