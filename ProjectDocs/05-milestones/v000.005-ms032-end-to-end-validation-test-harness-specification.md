# MS032 — End-to-End Validation — Test Harness Specification

---

## Purpose

Define a deterministic, reproducible test harness for validating the full PhotoForge pipeline end-to-end.

This specification operationalizes MS032 by defining:

- fixture structure
- golden output format
- validation scripts
- comparison rules
- execution model

This document introduces no changes to pipeline behavior.

---

## Scope

Applies to:

- end-to-end validation only
- test harness structure
- validation execution

Does NOT modify:

- scanner
- planner
- contextual grouping logic
- reporter
- model structures

---

## Test Harness Structure

All test harness assets must be stored under:

```sh
tests/e2e/
```

### Directory Layout

```tree
tests/e2e/
├── fixtures/
├── golden/
├── manifests/
├── helpers.py
├── test_e2e_baseline.py
├── test_e2e_context.py
├── test_e2e_determinism.py
├── test_e2e_edge_cases.py
```

---

## Fixtures

### Definition

A fixture is a deterministic input dataset stored as a filesystem tree.

### Rules

- Fixtures must be stored under:

  ```sh
  tests/e2e/fixtures/<fixture_id>/input/
  ```

- Fixtures must be:
  - static
  - version-controlled
  - immutable during test execution
- Tests must copy fixtures to a temporary directory before execution
- Tests must never operate directly on fixture directories
- Fixtures must not depend on:
  - system time
  - environment variables
  - external services
  - locale
  - filesystem ordering

---

## Fixture Set (Minimum)

The following fixtures are required:

- `empty`
- `single-jpeg`
- `duplicate-pair`
- `identical-timestamps`
- `timestamp-gap`
- `corrupt-and-unsupported`
- `mixed-realistic`

Each fixture must represent a distinct validation scenario defined in MS032.

---

## Manifests

Each fixture must have a manifest file:

```sh
tests/e2e/manifests/<fixture_id>.json
```

### Format

```json
{
  "fixture_id": "<fixture_id>",
  "input_dir": "tests/e2e/fixtures/<fixture_id>/input",
  "scenarios": {
    "default": true,
    "context": true,
    "determinism_runs": 3
  }
}
```

### Manifests Rules

- Manifest defines:
  - which scenarios to execute
  - number of determinism runs
- Manifest must not encode implementation logic

---

## Golden Outputs

Golden outputs define the expected results for each fixture.

### Location

```sh
tests/e2e/golden/<fixture_id>/
```

### Required Files

- `default.console.txt`
- `default.json`
- `context.console.txt`
- `context.json`
- `models.json`

---

## Golden Output Rules

### General

- Golden outputs must be:
  - deterministic
  - version-controlled
  - exact reference outputs

### Console Output

- Must match exact stdout
- Must include whitespace and ordering
- Compared using exact string equality

### JSON Output

- Must match exact serialized output
- No structural comparison allowed

### Model Output (`models.json`)

Must contain:

- serialized `ScanResult`
- serialized `ContextualGrouping`
- serialized `PlanResult`

---

## Serialization Rules

All serialized outputs must follow strict rules:

- Encoding: UTF-8
- Line endings: LF only
- Indentation: 2 spaces
- Object keys: sorted
- Datetime format: fixed canonical format
- Tuples: serialized as arrays
- Paths:
  - must be normalized using placeholder substitution

---

## Path Normalization

Absolute paths must be replaced with deterministic placeholders:

- `<INPUT_ROOT>`
- `<OUTPUT_ROOT>`

Example:

```sh
/tmp/run123/input/a.jpg
```

becomes:

```sh
<INPUT_ROOT>/a.jpg
```

Rules:

- Only test harness may perform normalization
- Production code must remain unchanged

---

## Validation Scripts

All scripts must be stored under:

```sh
scripts/
```

### 1. generate_e2e_golden.py

Purpose:

- generate golden outputs

Rules:

- must require explicit `--write`
- must use real pipeline execution
- must overwrite all golden files atomically

---

### 2. verify_e2e_golden.py

Purpose:

- validate outputs against golden files

Rules:

- must:
  - execute all scenarios
  - compare outputs using exact string equality
- must fail on any mismatch

---

### 3. list_e2e_fixtures.py

Purpose:

- list all registered fixtures

Rules:

- must output fixture IDs in deterministic order

---

## Test Modules

### test_e2e_baseline.py

Validate:

- default console output equals golden
- default JSON output equals golden
- no contextual grouping present

---

### test_e2e_context.py

Validate:

- contextual output present when enabled
- output matches golden files
- baseline sections unchanged

---

### test_e2e_determinism.py

Validate:

- identical outputs across multiple runs
- identical:
  - FileRecord sets
  - ContextualGrouping
  - PlanResult
  - console output
  - JSON output

Must invoke:

- `validate_contextual_grouping(...)`
- `validate_contextual_group(...)`

---

### test_e2e_edge_cases.py

Validate all edge cases defined in MS032.

---

## Dataset Stability Requirement

- Fixture datasets must remain unchanged across runs
- Filesystem state must not change during validation
- Any mutation invalidates determinism validation

---

## Golden Update Policy

- Golden files must only be updated when:
  - behavior change is intentional
  - change is defined in an approved milestone
- Golden updates must be reviewed explicitly
- Unintended diffs must be treated as failures

---

## Constraints

- No modification of pipeline logic
- No modification of model structures
- No duplication of validation logic outside model validators
- No tolerant comparisons
- No heuristic behavior

---

## Acceptance Criteria

- All fixtures execute successfully
- All outputs match golden files exactly
- Determinism is confirmed across runs
- Model invariants are validated using existing functions
- No regressions are detected
- No unintended behavior is introduced

---

## Success Definition

The test harness provides:

- deterministic validation of the full pipeline
- exact regression detection
- reproducible results across environments
- strict enforcement of all MS032 requirements
