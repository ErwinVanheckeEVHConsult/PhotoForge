# Milestone — v000.005-ms032 — end-to-end-validation

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic pipeline for scanning, hashing, duplicate grouping, planning, and reporting
- v0.5 introduced:
  - metadata abstraction and normalization (MS024–MS026)
  - pipeline metadata integration (MS027)
  - contextual grouping model (MS028)
  - contextual grouping rules (MS029)
  - contextual grouping integration (MS030)
  - reporter contextual output (MS031)
- The full pipeline now includes:
  - scanner → produces `FileRecord`
  - contextual grouping → produces `ContextualGrouping`
  - planner → produces `PlanResult`
  - reporter → produces console and JSON output
- All components are individually validated
- No full end-to-end validation has yet been performed across the complete pipeline

---

## Goal

Validate the entire pipeline end-to-end to ensure:

- strict determinism
- structural correctness
- absence of unintended side effects
- full alignment with all milestone contracts

---

## Responsibilities

- Execute full pipeline using the orchestration function
- Validate consistency between all pipeline stages
- Validate determinism across repeated runs
- Validate that all invariants defined in previous milestones are preserved
- Validate that contextual grouping integration and reporting behave correctly in combination

---

## Deliverables

- `End-to-end validation execution`
- `Determinism validation across multiple runs`
- `Validation of model invariants across all pipeline outputs`
- `Validation of reporter output (default and contextual modes)`
- `No unintended behavior across pipeline integration`

---

## Behavior

### Required behavior

- Validation must use the orchestration function introduced in MS030
- Validation must use real filesystem inputs (no mocked pipeline)
- Validation must cover:
  - scanning
  - metadata normalization
  - contextual grouping
  - planning
  - reporting (console and JSON)

---

## Baseline Definition

- A baseline output must be captured from the implementation **before enabling contextual output**
- Baseline must include:
  - console output
  - JSON output
- Baseline outputs must be stored locally as reference files
- Baseline must be generated using:
  - identical input dataset
  - identical CLI invocation (without `--context`)

---

## Baseline Comparison Rules

- Baseline comparison must use **exact string equality**
- Console output comparison must be:
  - exact string match (including whitespace and ordering)
- JSON output comparison must be:
  - exact string match after serialization
  - including ordering and formatting
- No tolerant or structural comparison is allowed
- Any difference must be treated as a failure

---

## Validation Scenarios

### 1. Baseline (default behavior)

- Run pipeline without `--context`
- Validate:
  - console output matches baseline exactly
  - JSON output matches baseline exactly
  - no contextual grouping appears in output

---

### 2. Contextual output enabled

- Run pipeline with `--context`
- Validate:
  - contextual grouping is present in console output
  - contextual grouping is present in JSON output
  - existing output remains unchanged relative to baseline
  - ordering of groups and members is deterministic

---

### 3. Determinism validation

- Run pipeline multiple times on identical input
- Validate:
  - identical `FileRecord` sets
  - identical `ContextualGrouping`
  - identical `PlanResult`
  - identical console output
  - identical JSON output

---

### 4. Edge cases

- Empty directory
- Single file
- Multiple files with identical timestamps
- Files with timestamp gaps exceeding grouping threshold
- Presence of corrupt files
- Presence of unsupported files

Validate:

- no crashes
- correct classification
- correct grouping behavior
- correct reporting output

---

## Invariant Validation

Invariant validation must use existing model validation mechanisms:

- `validate_contextual_grouping(...)`
- `validate_contextual_group(...)`
- dataclass `__post_init__` validation logic

Rules:

- Validation functions must be explicitly invoked
- No custom or duplicated validation logic is allowed
- All invariants must be validated on:
  - `ContextualGrouping`
  - all contained `ContextualGroup` objects

---

## Non-Regression Validation

- Default (non-context) output must match baseline exactly
- No differences are allowed in:
  - console output
  - JSON output
- Contextual output must only appear when `--context` is enabled

---

## Constraints

- No changes to implementation logic are allowed in this milestone
- No modifications to scanner, planner, grouping, or reporter logic
- No modification of model structures
- Validation must not introduce new behavior
- No redesign introduced

---

## Acceptance Criteria

- All validation scenarios complete successfully
- Baseline outputs are captured and stored
- Baseline comparison uses exact string equality
- Determinism is confirmed across repeated runs
- No differences are observed between identical runs
- Default output matches baseline exactly
- Contextual output appears only when explicitly enabled
- All model invariants are validated using existing validation functions
- No unintended side effects occur
- No regressions are detected

---

## Risks

- Hidden non-determinism in ordering or formatting
- Undetected coupling between grouping and planning
- Reporter output inconsistencies
- Edge case failures not covered by scenarios

---

## Success Definition

- The full pipeline is validated as deterministic, stable, and correct
- All milestone contracts are satisfied in combination
- The system is ready for release
