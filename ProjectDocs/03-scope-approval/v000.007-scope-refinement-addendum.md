# Scope Refinement Addendum — v000.007

This document refines the approved v0.7 scope by defining all behavior in a deterministic, unambiguous manner.

This document does not expand scope.
It only removes ambiguity and makes all scoped items requirement-ready.

---

## 1. Timestamp Resolution Model

### 1.1 Source Precedence (strict order)

Timestamp must be resolved using the following ordered sources:

1. EXIF timestamp (with timezone if available)
2. Filename-derived timestamp
3. Folder name-derived timestamp
4. Filesystem timestamp (mtime)

The first valid source in this order must be used.

---

### 1.2 Validity Rules

A timestamp source is considered **valid** only if:

* It can be parsed deterministically
* It contains all required components:

  * date (year, month, day)
  * time (hour, minute, second)
* It does not violate basic calendar constraints

If a source is invalid, it must be ignored and the next source used.

Presence of an invalid higher-precedence source must not block fallback.

---

### 1.3 Conflict Handling

If multiple valid timestamps exist across sources:

* Only the highest-precedence source is used for resolution
* Lower-precedence sources are ignored for resolution
* Differences between sources must be reported via diagnostics (see section 4)

No automatic correction or reconciliation is performed.

---

### 1.4 Timezone Handling

#### 1.4.1 EXIF timezone present

If EXIF provides timezone information:

* The timestamp must be interpreted using that timezone
* The timestamp must be normalized to UTC

#### 1.4.2 EXIF timezone absent

If EXIF timestamp exists but no timezone is present:

* The timestamp must be treated as **naive**
* No implicit timezone inference is allowed
* No conversion is performed

#### 1.4.3 Non-EXIF sources

* Filename, folder, and filesystem timestamps are always treated as naive
* No timezone inference is allowed

---

## 2. Timestamp Representation Model

Each timestamp source must be represented using a structured model exposing multiple representations.

### 2.1 Representations

For each valid timestamp source, the following fields must be defined:

* `naive_timestamp`
* `timezone_offset` (if available)
* `aware_timestamp` (only if timezone is available)
* `utc_timestamp` (only if timezone is available and conversion is possible)

### 2.2 Representation Rules

* `naive_timestamp` must always be present for valid timestamps
* `timezone_offset` must be present only if explicitly provided by the source
* `aware_timestamp` must exist only if both timestamp and timezone are available
* `utc_timestamp` must exist only if `aware_timestamp` exists
* No implicit timezone inference is allowed

---

### 2.3 Comparison Rules

Two timestamp sources are directly comparable only if they share a common representation:

* UTC timestamps are comparable with UTC timestamps
* Naive timestamps are comparable with naive timestamps
* Otherwise, they are not directly comparable

If timestamp sources are not directly comparable, no inconsistency diagnostic is produced.

Inconsistency must be defined only between directly comparable timestamps.

---

## 3. Metadata Source Definitions

### 3.1 EXIF Timestamp Fields

The following EXIF fields are in scope:

* DateTimeOriginal (primary)
* DateTimeDigitized (fallback)
* DateTime (fallback)

Priority within EXIF:

1. DateTimeOriginal
2. DateTimeDigitized
3. DateTime

The first valid field must be used.

---

### 3.2 EXIF Timezone Fields

If present, the following must be used:

* OffsetTimeOriginal
* OffsetTimeDigitized
* OffsetTime

Timezone must be applied only to the corresponding timestamp field.

---

### 3.3 Filename Timestamp Extraction

#### 3.3.1 Supported Patterns

Full datetime patterns:

* YYYYMMDD_HHMMSS
* YYYY-MM-DD_HH-MM-SS
* YYYY-MM-DD HH-MM-SS

Date-only patterns:

* YYYYMMDD
* YYYY-MM-DD

---

#### 3.3.2 Rules

* Pattern matching must be deterministic
* Parsing must be left-to-right, first full match wins
* Full datetime patterns have priority over date-only patterns
* If multiple matches of the same priority exist:

  * the first match in filename order is used
* Matches must respect character boundaries (no partial substring extraction)
* Invalid parses must be rejected
* Match order is determined by left-to-right scanning of the raw string.

For date-only patterns:

* Time defaults to 00:00:00

---

## 3.4 Folder Name Timestamp Extraction

### 3.4.1 Supported Patterns

Single date patterns:

* YYYYMMDD
* YYYY-MM-DD
* YYYY MM DD

Range patterns:

* YYYY MM DD - DD
* YYYY MM DD - MM DD
* YYYY MM DD - YYYY MM DD
* YYYY-MM-DD - DD
* YYYY-MM-DD - MM-DD

---

### 3.4.2 Rules

* Folder parsing applies only to the immediate parent folder
* Pattern matching must be deterministic
* Parsing must be left-to-right, first full match wins
* If multiple matches exist:

  * the first match in folder name order is used
* Matches must respect character boundaries
* Invalid parses must be rejected
* Match order is determined by left-to-right scanning of the raw string.

For range patterns:

* Only the start date is used
* End date is ignored

For all folder-derived timestamps:

* Time defaults to 00:00:00

---

## 4. Metadata Diagnostics Model

Diagnostics are **observational only** and must not affect pipeline behavior.

---

### 4.1 Diagnostic Categories

Each file may produce zero or more diagnostics:

#### 4.1.1 Missing

* No valid timestamp found in any source

#### 4.1.2 Invalid

* A source contains a timestamp but fails validation
* Invalid sources are ignored for resolution but still reported

#### 4.1.3 Inconsistent

* Multiple valid timestamp sources exist
* Sources are directly comparable
* Their values differ

---

### 4.2 Diagnostic Rules

* Diagnostics must be deterministic
* Diagnostics must be reproducible across runs
* Diagnostics must not depend on environment
* Diagnostics must not alter resolution behavior

---

## 5. Format Scope Clarification

### 5.1 Supported Formats

The following formats are in scope:

* All formats currently supported by the pipeline
* TIFF

---

### 5.2 Processing Rules

* All supported formats must pass through the same pipeline stages
* No format-specific branching is allowed outside metadata extraction
* Metadata normalization must produce identical structures across formats

---

### 5.3 Implementation Constraint

* Each supported format must have a corresponding metadata extractor
* Metadata extractors must conform to the metadata_extractors structure

---

## 6. Filesystem Timestamp Definition

* Filesystem timestamp refers to file modification time (mtime)
* It must be read in UTC
* It must be truncated to second precision

---

## 7. Determinism Guarantees

All behavior defined in this document must ensure:

* identical input → identical output
* no implicit ordering
* no environment-dependent behavior
* no fallback ambiguity

---

## 8. Non-Goals (Reaffirmed)

* No GPS-based timezone inference
* No interactive correction
* No duplicate handling changes
* No performance optimization
* No CLI or UX changes
* No cross-file timestamp correction or inference
* No device-based trust model
