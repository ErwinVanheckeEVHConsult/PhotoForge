# Milestone — v000.005-ms026 — metadata-normalization

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic JPEG-only pipeline
- v0.5 introduces multi-format support, metadata abstraction, and contextual grouping
- MS022 introduced detection of additional formats
- MS023 introduced deterministic validation boundaries
- MS024 introduced a passive metadata abstraction layer with:
  - `timestamp`
  - `timestamp_source`
- MS025 introduced format-specific metadata extraction for non-JPEG formats, remaining isolated from the active pipeline
- Existing JPEG extraction behavior remains the reference behavior and must not change
- Metadata extracted across formats must be normalized before any pipeline integration

---

## Goal

Ensure that metadata extracted from all supported formats is normalized into a consistent, deterministic structure without introducing any behavioral changes to the active pipeline.

---

## Responsibilities

- Define and enforce normalization rules for metadata across all formats
- Ensure all extracted metadata conforms to the normalized metadata structure
- Ensure consistency of:
  - `timestamp`
  - `timestamp_source`
- Preserve existing JPEG metadata behavior exactly
- Prepare normalized metadata for safe pipeline integration in later milestones
- Introduce a normalization entry point for metadata

---

## Deliverables

- `Normalization rules for metadata across formats`
- `Consistent representation of timestamp values`
- `Consistent representation of timestamp_source values`
- `Validation that all extracted metadata conforms to the normalized structure`
- `No changes to active pipeline behavior`
- `Central normalization function enforcing metadata structure`

---

## Behavior

### Required behavior

- All metadata produced by extraction functions must conform to the normalized metadata structure:
  - `timestamp: datetime`
  - `timestamp_source: str`
- Timestamp values must:
  - be deterministic for identical input
  - follow the same normalization rules as JPEG extraction
  - remain naive (no timezone conversion)
- `timestamp_source` values must:
  - be explicit
  - be consistent across formats
  - use deterministic naming
- JPEG metadata must remain unchanged:
  - same timestamp values
  - same timestamp_source values
- No additional metadata fields are allowed
- No metadata inference or enrichment is allowed
- No fallback logic may be introduced beyond what is explicitly defined in format-specific extraction
- No normalization logic may alter existing values for JPEG files
- Normalization must be applied through a central normalization function in `metadata.py`
- Format-specific extraction logic must remain outside `metadata.py`
- Extraction functions must not duplicate normalization behavior

### Normalization Rules

- Datetime values must be valid Python `datetime` objects
- No timezone normalization or conversion is allowed
- Invalid or missing values must not be corrected heuristically
- Metadata must be representable identically across formats

### Rules

- Deterministic behavior only
- No implicit or heuristic decisions
- No branching-based behavior changes outside defined normalization rules
- Normalization must be structural, not interpretative

---

## Constraints

- No changes to scanner behavior
- No changes to hashing logic
- No changes to duplicate grouping
- No changes to planner behavior
- No changes to reporter behavior
- No modification of `FileRecord` fields or structure
- No modification of existing JPEG extraction logic
- No integration of non-JPEG metadata into the active pipeline
- No pipeline restructuring
- No additional metadata fields beyond `timestamp` and `timestamp_source`
- No metadata inference, enrichment, or correction
- Normalization logic must be centralized in `metadata.py`
- No format-specific extraction logic may be implemented in `metadata.py`
- No redesign of abstraction layer

---

## Acceptance Criteria

- All metadata produced by extraction functions conforms to the normalized structure
- Timestamp values are consistent and deterministic across formats
- `timestamp_source` values are consistent and deterministic across formats
- JPEG metadata values remain identical to prior behavior
- No additional metadata fields are introduced
- No changes occur in scanner output
- No changes occur in `FileRecord`
- No changes occur in downstream pipeline behavior
- Normalization introduces no implicit or branching-based behavior changes
- Normalization is performed exclusively through the central normalization function in `metadata.py`

---

## Risks

- Inconsistent timestamp representations across formats
- Accidental normalization that alters JPEG values
- Hidden introduction of heuristics or implicit corrections
- Divergent `timestamp_source` naming across formats
- Leakage of normalization logic into extraction or pipeline layers

---

## Success Definition

- Metadata across all formats is normalized into a consistent structure
- JPEG behavior remains strictly identical
- No unintended side effects are introduced
- The system is ready for safe integration of metadata into the pipeline
