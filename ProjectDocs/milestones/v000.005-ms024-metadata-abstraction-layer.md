# Milestone — v000.005-ms024 — metadata-abstraction-layer

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic pipeline for JPEG processing
- Metadata extraction is currently embedded in EXIF-specific logic through `extract_timestamp(...)`
- Current JPEG metadata extraction returns only:
  - `timestamp`
  - `timestamp_source`
- `FileRecord` already stores `timestamp` and `timestamp_source` and this structure must not change
- Only JPEG files are processed in existing versions
- v0.5 introduces multi-format support and therefore requires a format-agnostic metadata representation layer
- MS022 introduced detection of additional formats without processing them
- Existing behavior must remain unchanged for JPEG files

---

## Goal

Introduce a format-agnostic metadata abstraction layer that separates normalized metadata representation from format-specific extraction, without changing any existing behavior.

---

## Responsibilities

- Define a normalized metadata structure for pipeline-relevant metadata
- Introduce a metadata abstraction layer that is structurally separate from format-specific extraction
- Preserve existing EXIF-based JPEG extraction behavior exactly
- Prepare the codebase for later multi-format metadata extraction milestones
- Ensure no downstream pipeline structure or behavior changes

---

## Deliverables

- `Normalized metadata structure definition`
- `Metadata abstraction layer introduced`
- `JPEG metadata remains behaviorally identical`
- `No change to FileRecord structure`
- `No change to scanner, planner, reporter, or hashing behavior`

---

## Behavior

### Required behavior

- Metadata extraction for JPEG files must behave exactly as before
- Existing EXIF fallback chain must remain unchanged:
  1. EXIF DateTimeOriginal
  2. EXIF DateTimeDigitized
  3. EXIF DateTime
  4. file modification time
- Timestamp values must remain identical for identical input
- `timestamp_source` values must remain identical for identical input
- The abstraction layer must define only the metadata currently required by the pipeline:
  - `timestamp: datetime`
  - `timestamp_source: str`
- No additional metadata fields are allowed in this milestone
- No metadata is extracted for non-JPEG formats in this milestone
- The abstraction layer must not contain format-specific extraction logic
- Format-specific logic must remain outside the abstraction layer

### Extraction Boundary

- Existing `extract_timestamp(...)` must remain unchanged
- The abstraction layer must not modify or replace EXIF extraction logic
- A wrapper may adapt the output of `extract_timestamp(...)` into the normalized metadata structure
- Format-specific extraction must remain outside the abstraction layer

### Rules

- Deterministic behavior only
- No implicit or heuristic decisions
- No metadata inference or enrichment
- No behavioral changes to JPEG extraction
- No downstream exposure of new metadata fields
- The abstraction layer must be structural, not behavioral

---

## Constraints

- No changes to scanner behavior
- No changes to hashing logic
- No changes to duplicate grouping
- No changes to planner behavior
- No changes to reporter behavior
- No modification of `FileRecord` fields or structure
- No modification of the existing EXIF extraction logic in `extract_timestamp(...)`
- No processing of non-JPEG formats
- No pipeline restructuring
- No behavioral changes introduced into the active pipeline
- Any abstraction layer introduced must be passive and must not alter execution flow or results
- No scanner integration with the abstraction layer in this milestone
- No premature metadata fields beyond `timestamp` and `timestamp_source`

---

## Acceptance Criteria

- A normalized metadata structure exists and is clearly defined
- The normalized metadata structure contains exactly:
  - `timestamp`
  - `timestamp_source`
- The abstraction layer is clearly separated from format-specific extraction code
- Existing JPEG metadata extraction produces identical `timestamp` results as before
- Existing JPEG metadata extraction produces identical `timestamp_source` results as before
- No changes occur in scanner output for JPEG files
- No changes occur in `FileRecord`
- No changes occur in downstream pipeline behavior
- No non-JPEG metadata extraction is introduced
- The abstraction layer introduces no implicit behavior and no branching-based behavior changes

---

## Risks

- Accidental modification of EXIF extraction behavior
- Introducing unused metadata fields prematurely
- Leakage of abstraction into runtime pipeline behavior
- Hidden coupling between abstraction and extraction logic
- Structural changes that silently alter scanner behavior

---

## Success Definition

- Metadata representation is structurally decoupled from format-specific extraction
- JPEG processing remains strictly identical
- The active pipeline remains unchanged
- The codebase is prepared for later multi-format metadata extraction without introducing premature behavior
