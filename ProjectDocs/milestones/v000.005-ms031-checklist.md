## Validation

- Reporter accepts `ContextualGrouping` as an additional input
- Reporter accepts `include_context: bool` as an additional input
- Reporter does not compute or modify contextual grouping
- Reporter treats contextual grouping as read-only input
- Contextual grouping does not alter:
  - existing report structure
  - existing summary values
  - existing action output
- Reporter function signatures are extended to accept:
  - `contextual_grouping: ContextualGrouping | None`
  - `include_context: bool`
- Reporter interface is defined as:
  - `render_console_report(plan_result, contextual_grouping=None, include_context=False)`
  - `render_json_report(plan_result, contextual_grouping=None, include_context=False)`
- Existing reporter functions remain the canonical rendering entry points
- No duplicate contextual-only reporter functions are introduced
- When `include_context` is `False`, reporter output remains strictly identical to prior behavior
- When `include_context` is `True`, `contextual_grouping` is provided
- Reporter does not infer contextual grouping presence from any source other than explicit function arguments
- Default console output remains strictly unchanged
- Default JSON output remains strictly unchanged
- Contextual grouping does not appear unless explicitly enabled
- A new CLI flag `--context` is introduced
- CLI calls the orchestration function introduced in the previous milestone
- CLI receives exactly:
  - `PlanResult`
  - `ContextualGrouping`
- CLI passes both objects to reporter functions
- CLI passes `include_context=True` only when `--context` is provided
- CLI passes `include_context=False` when `--context` is not provided
- CLI does not compute, transform, or validate contextual grouping
- CLI does not render contextual grouping directly
- CLI remains a propagation layer only
- When `include_context=True`, console output includes a separate contextual grouping section
- The contextual grouping section appears after existing report output
- Each contextual group includes:
  - `group_id`
  - ordered list of `record_ref` values
- Console output is deterministic
- Groups are printed in the exact order defined by `ContextualGrouping.groups`
- Members are printed in the exact order defined in `member_refs`
- No additional formatting or interpretation is introduced
- Existing console output content remains unchanged outside the added contextual section
- When `include_context=True`, JSON output includes the top-level field:
  - `contextual_groups`
- Each contextual group in JSON is represented as:
  - `group_id`
  - `member_refs`
- Order of groups in JSON matches `ContextualGrouping.groups`
- Order of `member_refs` in JSON is preserved
- No additional fields are introduced in contextual JSON output
- Existing JSON schema remains unchanged otherwise
- When `include_context=False`, the `contextual_groups` field is not present
- For identical input:
  - identical `PlanResult` and identical `ContextualGrouping` produce identical output
- Output does not depend on:
  - scan order
  - filesystem traversal
  - runtime environment
  - CLI invocation order
- Reporter does not:
  - recompute grouping
  - validate grouping beyond structural correctness
  - interpret grouping semantics
- Reporter does not influence:
  - planner
  - pipeline orchestration
  - grouping logic
- No unintended side effects occur
- No existing output fields are modified or removed
