---

## Validation

- The contextual grouping model is clearly defined and documented
- The contextual group structure is explicitly defined
- The model defines how contextual groups are represented
- The model defines how `FileRecord` objects are associated with contextual groups
- Group membership is defined structurally (association only), not behaviorally
- The model supports representing a partition of `FileRecord` objects
- The model allows each `FileRecord` to be associated with exactly one contextual group
- The model prevents association of a `FileRecord` with multiple contextual groups
- The method by which groups are constructed is not defined in this milestone
- No grouping rules are defined
- No grouping thresholds are defined
- No grouping execution logic is defined
- No ordering-based grouping logic is defined
- The model does not depend on filesystem traversal order
- The model does not depend on scan order
- The model does not depend on external state or environment
- The model allows future grouping logic to use only explicitly allowed metadata:
  - `timestamp`
  - `timestamp_source`
- No additional metadata fields are introduced
- No implicit or heuristic grouping behavior is introduced
- No probabilistic or AI-based grouping concepts are introduced
- No undocumented assumptions about grouping are introduced
- Contextual grouping is clearly separated from duplicate grouping
- Contextual grouping does not interact with or modify:
  - duplicate grouping
  - canonical selection
  - planner behavior
  - reporter behavior
- The model does not modify or wrap `FileRecord` objects
- The model references existing `FileRecord` instances only
- No unintended side effects
- Deterministic behavior is preserved
- `record_ref` is defined as the normalized absolute file path represented as a string
- `record_ref` is derived deterministically from FileRecord.path
- `group_id` is defined as a deterministic identifier derived only from member_refs
- `group_id` is computed using canonical JSON encoding and SHA-256
- `group_id` is stable for identical member_refs
- The encoding used for group_id is unambiguous and delimiter-safe
- member_refs are required to be sorted lexicographically
- contextual groups are required to be sorted lexicographically by group_id
- ordering is explicitly defined and independent of scan or filesystem traversal
- Each contextual group must contain at least one member
- member_refs within a group must be unique
- A record_ref must not appear in more than one contextual group
- ContextualGroup and ContextualGrouping are immutable structures
- If no FileRecord objects exist, the grouping is represented as an empty tuple of groups
