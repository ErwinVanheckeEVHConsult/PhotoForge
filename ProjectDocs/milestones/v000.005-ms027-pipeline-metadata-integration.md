# Milestone — v000.005-ms027 — pipeline-metadata-integration

Starting milestone.

---

## Context

- v0.1–v0.3 define a deterministic JPEG-only pipeline
- v0.5 introduces multi-format support, metadata abstraction, and contextual grouping
- MS022 introduced detection of additional formats
- MS023 introduced deterministic validation boundaries
- MS024 introduced a passive metadata abstraction layer (`NormalizedMetadata`)
- MS025 introduced format-specific metadata extraction for non-JPEG formats (still isolated)
- MS026 introduced centralized metadata normalization in `metadata.py`
- Current scanner pipeline:
  - reads file metadata (stat)
  - extracts timestamp via `extract_timestamp(...)` (JPEG only)
  - computes SHA-256
  - produces `FileRecord`
- Non-JPEG formats are still not part of the active pipeline
- Metadata extraction and normalization are implemented but not yet integrated

---

## Goal

Integrate the metadata abstraction and normalization layers into the active pipeline so that all currently processable files (JPEG only in this milestone) use a unified, normalized metadata flow without changing existing behavior.

---

## Responsibilities

- Integrate metadata normalization into the scanner pipeline
- Route metadata through the central normalization function
- Preserve existing JPEG behavior exactly
- Ensure integration does not alter pipeline structure or outputs
- Prepare the pipeline for safe multi-format processing in later milestones

---

## Deliverables

- `Metadata normalization integrated into scanner pipeline`
- `Central normalization function used for all metadata entering the pipeline`
- `No behavioral changes to existing JPEG processing`
- `Pipeline prepared for multi-format metadata consumption`

---

## Behavior

### Required behavior

- All metadata used to construct `FileRecord` must pass through the central normalization function in `metadata.py`
- JPEG files must:
  - still use `extract_timestamp(...)`
  - produce identical `timestamp` values
  - produce identical `timestamp_source` values
- For JPEG inputs, normalization must act as an identity mapping (no transformation of `timestamp` or `timestamp_source`)  
- For JPEG inputs, the values returned by normalization must be strictly equal to the input values:
  - `normalized.timestamp == input.timestamp`
  - `normalized.timestamp_source == input.timestamp_source`
- No reconstruction, reformatting, or transformation of values is allowed
- The integration must not change:
  - file inclusion/exclusion behavior
  - hashing behavior
  - duplicate grouping
  - planner behavior
  - reporter behavior
- The normalization step must be applied consistently and deterministically
- No additional metadata fields may be introduced
- Timestamps must remain naive (`datetime` without timezone)
- No timezone inference or conversion is allowed
- The normalization function must not raise new exceptions for valid JPEG inputs
- The normalization function must not suppress or alter existing exception behavior from upstream logic

### Integration Rules

- Integration must be minimal and localized to the point where metadata is produced
- Existing EXIF extraction logic must not be modified
- The normalization function must be called immediately after metadata extraction
- In the scanner pipeline, normalization must be applied immediately after `extract_timestamp(...)` and before `FileRecord` construction
- No format-specific extraction logic may be introduced into the scanner
- The metadata abstraction layer must not introduce new behavior beyond normalization
- The scanner must call the normalization function exactly once per processed file
- No subsequent normalization calls on already normalized values are allowed
- The central normalization interface must be a single function in `metadata.py`
- The normalization function must accept exactly:
  - `timestamp: datetime`
  - `timestamp_source: str`
- The normalization function must return a `NormalizedMetadata` object
- `FileRecord` construction must use `timestamp` and `timestamp_source` taken directly from the returned `NormalizedMetadata` object
- No tuple-based, dict-based, or alternative normalization return paths are allowed
- Values returned by `extract_timestamp(...)` must not be modified before being passed to the normalization function

### Non-JPEG Handling

- Non-JPEG formats must remain non-processable in this milestone
- No `FileRecord` objects must be produced for non-JPEG formats
- No integration of non-JPEG extraction into the pipeline is allowed

### Rules

- Deterministic behavior only
- No implicit or heuristic decisions
- No branching-based behavior changes outside defined integration rules
- No change to existing execution flow beyond normalization insertion
- Direct construction of metadata used in `FileRecord` must not bypass the normalization function

---

## Constraints

- No changes to hashing logic
- No changes to duplicate grouping
- No changes to planner behavior
- No changes to reporter behavior
- No modification of `FileRecord` fields or structure
- No modification of `extract_timestamp(...)` behavior
- No integration of non-JPEG extraction into the pipeline
- No pipeline restructuring
- No metadata inference, enrichment, or correction
- No additional metadata fields beyond `timestamp` and `timestamp_source`
- Normalization must be centralized in `metadata.py`
- No format-specific extraction logic may be implemented in the scanner
- Normalized metadata values must not be modified after normalization and before `FileRecord` construction
- The values of `timestamp` and `timestamp_source` in all produced `FileRecord` objects must be identical to those produced prior to this milestone
- The normalization function must not introduce fallback behavior or error suppression
- No alternative normalization or metadata construction paths may be introduced
- No redesign introduced

---

## Acceptance Criteria

- All metadata used to construct `FileRecord` must be obtained directly from the output of the central normalization function in `metadata.py`
- `FileRecord` objects constructed after integration are byte-for-byte equivalent to those produced prior to this milestone
- The normalization function must be invoked in the scanner at the exact point where timestamp metadata is first produced
- No additional invocation points are allowed elsewhere in the pipeline
- The normalization function is applied to all metadata used in `FileRecord` construction without exception
- JPEG `timestamp` values remain identical to prior behavior
- JPEG `timestamp_source` values remain identical to prior behavior
- No changes occur in scanner output for identical input
- No changes occur in `FileRecord`
- No changes occur in downstream pipeline behavior
- For identical JPEG-only input, the following remain identical:
  - `records` (content and ordering)
  - `skipped` (content and ordering)
  - `issues` (content and ordering)
  - `supported_files_processed`
  - `total_entries_seen`
- Non-JPEG formats still do not produce `FileRecord` objects
- No additional metadata fields are introduced
- No implicit or branching-based behavior changes are introduced
- Deterministic behavior is preserved
- The normalization interface is explicit and unambiguous:
  - input: `timestamp: datetime`, `timestamp_source: str`
  - output: `NormalizedMetadata`
- No alternative normalization signatures or return types are introduced

---

## Risks

- Accidental modification of JPEG metadata values during normalization
- Hidden behavioral changes introduced during integration
- Leakage of format-specific logic into the scanner
- Inconsistent application of normalization across code paths
- Silent regression in pipeline determinism

---

## Success Definition

- Metadata normalization is consistently applied within the pipeline
- JPEG behavior remains strictly identical
- The pipeline remains deterministic and unchanged in behavior
- The system is ready for safe integration of non-JPEG formats in later milestones
