# Milestone — v000.005-ms023 — scanner-format-validation

Starting milestone.

---

## Context

- v0.1, v0.2, v0.3 are complete and released (alpha)
- v0.3 introduces deterministic corrupt file detection and reporting
- MS022 introduces detection of additional file formats (PNG, HEIC, RAW, video)
- Scanner currently fully processes JPEG files through the pipeline
- Newly detected formats are not yet processable by the pipeline
- Corrupt file classification is defined as failure during deterministic processing
- Existing corrupt classification behavior must remain unchanged

---

## Goal

Introduce deterministic format-aware validation in the scanner to distinguish between:

- processable files
- recognized but non-processable files
- corrupt files

without altering existing corrupt file semantics or pipeline behavior.

---

## Definitions

- **Processable format**: a format fully supported by the current pipeline (JPEG)
- **Recognized format**: a format detected by extension but not yet supported for processing
- **Corrupt file**: a file that fails deterministic processing according to v0.3 rules

---

## Responsibilities

- Introduce deterministic format classification based on file extension
- Introduce validation logic to distinguish processable, recognized, and corrupt files
- Ensure recognized formats are not misclassified as corrupt
- Integrate validation with existing corrupt file handling
- Preserve all existing behavior for JPEG processing

---

## Deliverables

- ``Deterministic format classification in scanner``
- ``Validation logic distinguishing processable, recognized, and corrupt files``
- ``Correct integration with existing corrupt file classification``

---

## Behavior

### Required behavior

Each file must be classified into exactly one of the following categories:

1. **Processable**
   - File extension corresponds to a processable format (JPEG)
   - File enters full processing pipeline:
     - metadata extraction
     - timestamp extraction
     - hashing
     - FileRecord creation

2. **Recognized but non-processable**
   - File extension corresponds to a recognized format introduced in MS022
   - File must:
     - not enter metadata extraction
     - not enter timestamp extraction
     - not enter hashing
     - not produce a FileRecord
   - File must be recorded deterministically as skipped with a specific reason

3. **Corrupt**
   - File is processable but fails deterministic processing
   - Includes:
     - I/O failure
     - metadata extraction failure
     - hashing failure
     - structurally invalid file content
   - Must follow existing v0.3 corrupt classification rules exactly

4. **Unsupported**
   - File extension is not recognized
   - File must not enter any processing step
   - File must be recorded deterministically as skipped with reason "unsupported_extension"

### Rules

- Format classification must be based solely on file extension
- No heuristic or probabilistic validation
- No content interpretation beyond validation scope
- No implicit classification rules
- Same file must always produce the same classification
- Recognized formats must never be classified as corrupt solely due to lack of processing support
- All files must fall into exactly one of the defined categories

---

## Constraints

- No metadata extraction changes
- No hashing changes
- No duplicate grouping changes
- No planner changes
- No reporter changes
- No pipeline restructuring
- Corrupt file semantics must remain consistent with v0.3 specification
- Recognized formats must remain fully excluded from the processing pipeline
- No partial processing of recognized formats is allowed

---

## Acceptance Criteria

- Format classification is deterministic and extension-based
- Processable files (JPEG) behave exactly as before
- Recognized formats are correctly excluded from processing
- Recognized formats are not misclassified as corrupt
- Corrupt files are identified strictly according to v0.3 rules
- No FileRecord is created for recognized, corrupt, or unsupported files
- No unintended side effects on downstream pipeline
- Deterministic behavior is preserved for identical inputs

---

## Risks

- Misclassification of recognized formats as corrupt
- Accidental partial processing of recognized formats
- Regression in existing corrupt file handling
- Introduction of non-deterministic validation behavior

---

## Success Definition

- Scanner deterministically separates processable, recognized, and corrupt files
- JPEG processing remains unchanged and fully deterministic
- Recognized formats are cleanly isolated from the pipeline
- Corrupt classification remains consistent with v0.3 specification
- No impact on downstream pipeline behavior
